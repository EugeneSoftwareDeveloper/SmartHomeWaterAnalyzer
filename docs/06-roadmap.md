# 06. Roadmap

Что планируется в `SmartHomeWaterAnalyzer`. Не жёсткие сроки, а приоритезированный список — что взять следующим, когда будет час-другой свободного времени.

История уже сделанного — в [`../CHANGELOG.md`](../CHANGELOG.md). Этот файл — только про **будущее**.

## Принципы приоритезации

Сортировка по убыванию пользы для пользователя:

1. **Высокий** — пользователь заметит на первом же замере. Чинит реальный баг, незакрытую функциональную дыру, или существенно ускоряет рабочий процесс.
2. **Средний** — польза проявляется после нескольких использований или при сравнении замеров между собой. Никто не страдает без этого, но «приятно».
3. **Низкий** — инженерная чистота, defensive coding, scale-out на других пользователей. Пользователь не заметит напрямую.
4. **Парковка** — идеи, которые звучат разумно, но могут оказаться не нужны. Перед реализацией стоит ещё раз спросить себя «а оно точно надо?».

## Высокий приоритет

### Trend indicators в карточках замера

**Что**: после ручного сохранения хорошо бы видеть, насколько изменилось значение относительно прошлого сохранённого замера. Электрод pH стабилизируется во времени, TDS реагирует на старение фильтра — без сравнения с прошлым каждое значение «висит в воздухе».

**Как закрыть**:
1. В `ReadingPage._refresh()` после успешного `readOnce` достать `HistoryRepository.recent(deviceId: ..., limit: 1)` — последнюю сохранённую запись.
2. Передать `previousValues` в `ParameterCard`.
3. Карточка показывает `↑ +0.12` / `↓ -0.05` маленьким текстом под значением. Цвет: зелёный при «улучшении» (зона стала лучше), красный при «ухудшении», серый при дельте < 2% от scaleMax (это шум).

**Готово, когда**: открыл приложение, прочитал → под каждым значением стоит «↑ +0.12 vs прошлый» с правильным цветом по направлению зоны.

### Реальные байты команд подсветки и HOLD

**Что**: текущие команды в [`lib/yinmik/commands.dart`](../lib/yinmik/commands.dart) — спекулятивная догадка по аналогии с BLE-YC01 (`0x08` = бит подсветки, `0x10` = бит HOLD). На большинстве партий BLE-C600 должно сработать, но если на твоей не сработало — переключатели в UI ничего не делают.

**Как закрыть**: два пути, любой работает.

- **Через приложение**: открой Reading → 🧪 в шапке (Debug-страница). Выбери целевой бит (`0x08` для подсветки или `0x10` для HOLD), нажми любой пресет, кнопкой «Send & verify» проверь — изменился ли бит в свежем кадре FF02. На успешный пресет приложение покажет зелёный ✓. После находки — впиши байты в [`lib/yinmik/commands.dart`](../lib/yinmik/commands.dart) и установи `areCommandsKnown = true`.
- **Через HCI snoop log с официального приложения**: пошаговая инструкция в [`03-control-commands.md`](./03-control-commands.md).

**Готово, когда**: переключатели в Reading-странице реально включают/выключают подсветку на приборе и HOLD-режим (бит в свежем кадре меняется как ожидается).

### Auto-reconnect к последнему устройству

**Что**: `AppSettings.lastDeviceId` уже сохраняется при каждом успешном подключении ([`lib/providers/app_settings.dart`](../lib/providers/app_settings.dart)). Не используется — после рестарта приложение начинает скан с нуля.

**Как закрыть**:
1. На HomePage добавить кнопку «Подключиться к [имя устройства]» сразу под appbar, если `lastDeviceId` не null.
2. По тапу — вызвать `FlutterBluePlus.connectedDevices` или попытаться `BluetoothDevice.fromId(lastDeviceId).connect(...)` напрямую, минуя скан.
3. Если коннект упал (прибор выключен / не рядом) — fallback на обычный скан с подсветкой именно этого MAC в списке.

**Готово, когда**: открыл приложение → один тап → читаю показания. Без скана и выбора из списка.

## Средний приоритет

### Режим сравнения замеров

**Что**: иногда хочется поставить два замера рядом — «было утром» и «сейчас», или «фильтр старый» и «фильтр новый» — и сразу увидеть, что изменилось.

**Как закрыть**:
1. На [`lib/ui/history_page.dart`](../lib/ui/history_page.dart) — long-press по записи → multi-select mode. Чекбоксы появляются у всех записей, заголовок меняется на «Выбрано: N».
2. Лимит 2–5 одновременно (больше не влезет в ширину экрана).
3. Кнопка «Сравнить» в шапке → новый экран `/history/compare?ids=[1,2,3]` → таблица: строки — параметры, столбцы — замеры. Цветовая индикация ячеек по зонам качества того же параметра.
4. Опционально: тот же график pH/etc, но с тремя линиями вместо одной.

**Готово, когда**: long-press → выбрал три замера → «Сравнить» → таблица с цветными ячейками.

### Pull-to-refresh на странице истории

**Что**: график обновляется реактивно через `recentMeasurementsProvider` стрим, так что технически refresh не нужен. Но pull-жест — привычка, без него UI ощущается «недо-mobile».

**Как закрыть**: обернуть `ListView` в `RefreshIndicator`, `onRefresh: () => ref.refresh(recentMeasurementsProvider.future)`.

**Готово, когда**: жест работает, спиннер крутится секунду, список перерисовывается.

### Кастомный фильтр имени из настроек

**Что**: после расширения фильтра в `YinmikBleClient.knownNameKeywords` ([`lib/yinmik/client.dart`](../lib/yinmik/client.dart)) приборы с экзотичными именами всё ещё могут не находиться. Это решается «Показать все устройства», но второй и третий запуск это раздражает.

**Как закрыть**:
1. В `AppSettings` добавить поле `customDeviceNameKeywords: List<String>` (persist через SharedPreferences).
2. На странице настроек — текстовое поле «Дополнительные имена прибора (через запятую)».
3. В `_looksLikeYinmik` — объединить hardcoded `knownNameKeywords` и кастомные из settings.

**Готово, когда**: пользователь вводит «MyWaterMeter» в настройках → приложение находит устройство с таким именем.

### Mock-based тесты HomePage / ReadingPage

**Что**: сейчас тесты — только декодер + Quality-логика. Виджеты — только smoke-тест `ColorGauge`. Если HomePage начнёт ломаться в новых версиях flutter_blue_plus, я узнаю об этом только при ручном тестировании.

**Как закрыть**:
1. `mocktail` уже в `dev_dependencies`.
2. Мокаем `YinmikBleClient` через `Provider.overrideWith` в `ProviderScope` тестового widget tree.
3. Сценарии: «нет разрешений» → видна ошибка, «скан вернул пустой список» → видна кнопка «Показать все устройства», «скан вернул прибор» → виден ListTile.

**Готово, когда**: 5–10 тестов вокруг HomePage state machine. CI-pipeline их прогоняет.

### Golden-тесты для виджетов карточек

**Что**: визуальная регрессия. Если кто-то случайно изменит цвет «danger» зоны или сдвинет padding в `ParameterCard`, тесты сейчас этого не поймают.

**Как закрыть**:
1. `flutter_test` поддерживает `matchesGoldenFile`.
2. Сделать golden для `ColorGauge` (одно изображение на каждую категорию), `ParameterCard` (свёрнутая + развёрнутая), `SummaryHeader`.
3. PR-pipeline проваливается, если pixel-diff больше 1%.

**Готово, когда**: в `test/widgets/goldens/` лежат PNG-ассеты, и `flutter test` их сравнивает.

## Низкий приоритет

### App icon + splash screen

**Что**: сейчас иконка — стандартный синий Flutter-флаг, splash — белый экран. Это нормально для personal-проекта, но если показать кому-то — выглядит «не доделано».

**Как закрыть**:
1. Нарисовать PNG 1024×1024 (например, капля воды + волна) — можно в Figma за 30 минут.
2. Положить в `assets/icon/icon.png` (путь уже прописан в pubspec для `flutter_launcher_icons`).
3. `dart run flutter_launcher_icons` — генерирует ассеты для всех размеров.
4. `dart run flutter_native_splash:create` — генерирует splash.

**Готово, когда**: на launcher вижу свою иконку, при старте приложения — свой splash, не стандартный Flutter.

### Sentry / Crashlytics

**Что**: `bootstrap.dart` сейчас в release-режиме молча гасит исключения, чтобы UI не падал. В debug-режиме пишет в консоль. Production crash — потеряется, узнаю о нём только из ревью «приложение крашится».

**Как закрыть**:
- Sentry: добавить `sentry_flutter` в dependencies, обернуть `runApp` в `SentryFlutter.init` с DSN из env.
- Crashlytics: добавить `firebase_crashlytics`, `FlutterError.onError = FirebaseCrashlytics.instance.recordFlutterError`.

DSN/конфиг — через `--dart-define=SENTRY_DSN=xxx` при сборке, чтобы не коммитить в репозиторий.

**Готово, когда**: специально сделать крэш (например, `throw Exception` в `onPressed`) → крэш виден в Sentry/Crashlytics dashboard.

### iOS-сборка

**Что**: Flutter-проект сконфигурирован под `--platforms=android`. Архитектура совместима с iOS (BLE, MaterialApp, SQLite — всё кросс-платформенное), но iOS-папка не создана.

**Как закрыть**:
1. `flutter create --platforms=ios .` в корне проекта.
2. В `ios/Runner/Info.plist` добавить:
   - `NSBluetoothAlwaysUsageDescription` — «Для подключения к тестеру качества воды».
   - `NSBluetoothPeripheralUsageDescription` — то же.
3. `flutter run -d <device>` на iPhone. Если permissions запрашиваются и сканер работает — готово.
4. На App Store — отдельная задача (нужны Apple Developer аккаунт за $99/год и подпись).

**Готово, когда**: APK работает на Android + IPA работает на iPhone. Один кодовый бэкенд.

### Фоновый опрос через workmanager

**Что**: для записи трендов «pH в течение дня» нужно автоматическое чтение, например, раз в час. Сейчас — только ручное.

**Как закрыть**:
1. `workmanager` пакет (платформенно интегрирован с Android WorkManager / iOS BGTask).
2. Periodic task с минимальным интервалом 15 минут (ограничение Android).
3. В task — `YinmikBleClient.readOnce` + `HistoryRepository.save`.
4. Уведомление, если значение out of range — переиспользовать `NotificationService`.

**Тонкости**:
- Прибор должен быть включён и в режиме BLE — пользователь не оставляет его постоянно работающим.
- Один BLE-канал — фоновый опрос конфликтует с открытым приложением и официальным YINMIK app.
- Battery drain — пользователь должен включать фон сознательно (опция в настройках).

**Готово, когда**: оставил прибор включённым в воде на ночь → утром в истории 8 точек через равные интервалы.

### Несколько устройств с alias-именами

**Что**: сейчас приложение помнит последнее одно устройство. Если у пользователя два прибора (например, дома и на даче), он каждый раз сканирует заново.

**Как закрыть**:
1. Заменить `lastDeviceId: String?` в `AppSettings` на `savedDevices: Map<String, String>` (MAC → alias).
2. На HomePage показывать список «Мои устройства» с alias-именами и иконкой коннекта.
3. По long-press — переименование / удаление.
4. Новое устройство добавляется автоматически при первом успешном подключении (диалог «Как назвать?»).

**Готово, когда**: на HomePage вижу «Кухня (BLE-C600)» и «Дача (BLE-C600)» — два моих прибора, между ними переключаюсь одним тапом.

## Парковка / отклонено

### Постоянное соединение с прибором

**Идея**: приложение держит BLE-соединение всегда, читает кадры с notifications вместо polling.

**Почему нет**: прибор портативный и работает от батарейки. Постоянное соединение разрядит его за несколько часов. Polling-модель «открыл приложение → читаю» — правильная для этого UX.

### Импорт UI / алгоритмов из WaterQualityApp

**Идея**: на GitHub есть open-source приложение WaterQualityApp для семейства BLE-YC01 с готовым UI и зонами для разных профилей. Скопировать оттуда.

**Почему нет**: оно под GPLv3. Наш проект — личный, лицензию ещё не выбирал, но GPL обязывает раскрывать весь код любого forka. Лучше переписать с нуля или взять MIT-friendly альтернативу.

### Web-версия

**Идея**: Flutter поддерживает Web target. Сделать веб-версию для просмотра истории с десктопа.

**Почему пока нет**: Web BLE API в браузерах фрагментирован (Chrome требует HTTPS + user gesture, Safari не поддерживает вообще), и сам прибор всё равно держит одно подключение — толку от веба мало. Если когда-нибудь сделаем cloud sync (см. ниже), вернёмся.

### Cloud sync

**Идея**: история синхронизируется через Firebase / iCloud между устройствами пользователя.

**Почему пока нет**: для personal-проекта избыточно. Можно ограничиться `share_plus` экспортом CSV (уже есть). Если приложение станет публичным — пересмотреть.

### Голосовой ассистент / Google Assistant Action

**Идея**: «Окей Google, какое pH воды».

**Почему нет**: out of scope для тестера качества воды. Слишком много инфраструктуры на единичный use-case.

## Как обновлять этот файл

- Когда задача из roadmap делается — переноси её в [`../CHANGELOG.md`](../CHANGELOG.md) (в раздел `[Unreleased]` → потом в релизный раздел) и удаляй отсюда.
- Новые идеи добавляй в правильный приоритет — не в конец списка.
- Если идея больше года висит в «Низком приоритете» без движения — скорее всего, ей место в «Парковке». Перенеси.
- Парковка — не помойка. Если идея отклонена, объясни **почему** одной строкой. Это полезно через год, когда возникнет соблазн вернуться.
